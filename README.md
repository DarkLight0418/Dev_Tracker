# Dev Tracker

개발 과정에서 발생하는 문제 → 해결 → 검증의 사고 흐름을 기록하고 추적 + 과정 중 피드백 개발 기록 시스템

## 프로젝트 목적

Dev Tracker는 결과물 중심의 개발 기록이 아닌,
개발자가 문제를 인식하고 해결해 나가는 전 과정을 구조적으로 남기기 위한 프로젝트입니다.

코드는 바뀌어도
의사결정과 사고 과정은 자산으로 남는다는 관점에서 출발합니다.

개인 프로젝트로서, 실력 향상을 목적으로 합니다.

## 해결하려는 문제
- 문제 해결 과정이 이슈, 커밋, 메모 등으로 분산됨
- 시간이 지나면 **왜 그렇게 구현했는지 설명할 수 없게 됨**
- 같은 실수를 반복하거나, 해결 경험을 재사용하기 어려움
- 개발 성장을 증명할 정제된 기록이 부족함

## 핵심 개념
Dev Tracker는 모든 기록을 다음 질문에 답하는 구조로 정리합니다.

1. 어떤 문제가 발생했는지
2. 왜 이 문제가 발생했는지
3. 어떤 해결 전략을 선택했는지
4. 실제 구현은 어떻게 이루어졌는지
5. 결과는 어땠는지
6. 다음에는 무엇을 개선할 수 있는지


## 기록 범위
Dev Tracker는 다음과 같은 개발 활동을 기록 대상으로 삼습니다.

- 버그 및 예외 상황 분석
- 기술/아키텍처 의사 결정
- API 및 데이터 구조 변경 이유
- 성능 및 보안, UX 개선 과정
- 실패한 시도와 그 판단 근거
- 개발 과정 중 느낀 점과 피드백 (개발자 로그)

> 결과만 기록하지 않고, **선택의 이유를 기록**합니다.

## 사용 대상
- 개인 개발자: 문제 해결 로그, 성장 기록, 포트폴리오 근거
- 팀 프로젝트: 기술 의사결정 공유, 온보딩 자료, 회고 기준

## 기본 원칙
- 문제 → 구현 → 검증 사이클을 유지한다
- 기술 기록은 재현 가능한 사실을 기준으로 작성한다
- 빠른 해결보다 이해 가능한 선택을 우선한다
- 실패한 시도도 기록 자산으로 남긴다

## 문서 구조 안내
상세 기록과 템플릿은 docs/ 디렉토리에서 관리합니다.
```
docs/
├─ problems/        # 문제 해결 기록 (기술 중심)
├─ decisions/       # 설계/의사결정 문서
├─ retrospectives/  # 주간/단계 회고
├─ logs/            # 개발자 로그 (감정·피드백·메타 기록)
├─ templates/       # 기록 템플릿
```

## 프로젝트 방향

> Dev Tracker는
“무엇을 만들었는가”보다
“왜 그렇게 만들었는가를 설명할 수 있는 개발자”를 목표로 합니다.


## 기술 스택
구체적인 기술 스택은 사용 패턴과 확장 요구에 따라 점진적으로 결정합니다.
