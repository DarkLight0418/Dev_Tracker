# Dev Tracker - Documentation Guide

이 문서는 Dev Tracker에서 생성되는 모든 기록의 **작성 기준과 품질을 유지하기 위한 내부 가이드**입니다.

README.md가 프로젝트의 방향을 선언한다면,  
이 문서는 그 방향을 **실제 기록으로 구현하기 위한 규칙**을 정의합니다.

---

## 1. 기록 분류 체계

Dev Tracker의 모든 기록은 다음 네 가지 유형 중 하나에 속해야 합니다.

### 1.1 Problems
문제 발생부터 해결 및 검증까지의 기술적 기록입니다.

- 버그
- 예외 상황
- 예상과 다른 동작

### 1.2 Decisions
기술적 선택과 그 근거를 기록합니다.

- 아키텍처 설계
- 라이브러리 선택
- 구조 변경

### 1.3 Logs
개발 과정 중 느낀 점과 판단의 맥락을 기록합니다.

- 감정
- 고민
- 직관
- 피드백

Logs는 기술 기록을 보조하는 역할이며,  
Problems나 Decisions를 대체하지 않습니다.

### 1.4 Retrospectives
일정 기간을 단위로 한 회고 기록입니다.

- 주간 회고
- 마일스톤 회고
- 프로젝트 단계 회고

---

## 2. 기록 품질 기준

모든 기록은 다음 기준을 충족해야 합니다.

- 기록 목적이 명확해야 한다
- 사실과 해석을 구분한다
- 재현 가능한 정보는 명시한다
- 결론 없이 방치하지 않는다

다음과 같은 기록은 지양합니다.

- 단순 감정 나열
- 결과 없는 메모
- 맥락이 없는 코드 조문

---

## 3. 기록 시점 기준

- Problems: 문제 인지 직후 초안 작성, 해결 후 보완
- Decisions: 선택 직후 또는 구현 전
- Logs: 개발 중 수시
- Retrospectives: 주기 종료 시

---

## 4. 기록 단위 기준

- 기록 1개는 하나의 질문에 답해야 한다
- 하나의 문서에 여러 문제를 혼합하지 않는다
- 작은 문제는 하나의 Problems 문서로 묶을 수 있다

---

## 5. 개발자 로그 작성 원칙

Logs에는 다음 내용이 포함될 수 있습니다.

- 당시의 감정 상태
- 판단에 영향을 준 요소
- 불안, 확신, 직감

단, Logs는 다음을 목표로 합니다.

> 감정을 기록하는 것이 아니라  
> **감정을 분석 가능한 데이터로 남기는 것**

---

## 6. 문서 간 관계 규칙

- Problems는 Decisions를 참조할 수 있다
- Logs는 Problems 또는 Decisions에 연결될 수 있다
- Retrospectives는 모든 기록을 종합한다

---

## 7. 공개 및 활용 기준

- Problems / Decisions: 포트폴리오 공개 가능
- Logs: 기본 비공개
- Retrospectives: 선택적 공개
